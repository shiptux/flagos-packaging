## 三、本会话回答的关键设计问题

1. **FlagTree 的 LLVM 来源**：OpenAI 官方 Azure Blob 上的预编译包
   （commit `10dc3a8e`，~LLVM 18 时代），构建时下载到 `~/.triton/llvm/`，
   不走发行版 LLVM —— 因为 Triton 跟随 LLVM 主干并依赖 MLIR 内部 API，
   发行版 LLVM 不可用。

2. **FlagTree 捆绑了什么**：除了 LLVM，还捆绑了 pybind11、NVIDIA `ptxas`、
   NVIDIA `cuobjdump`。后两者属 NVIDIA EULA 闭源，**导致包无法进入
   Debian/Ubuntu/Fedora 官方主仓库**，但作为 vendor 发行可以接受。已在
   `debian/copyright` 和 `plan-tracking.md` 中如实记录。

3. **一份源码同时打 C++ 和 Python 包**：可以。Debian 标准模式 —— 一个
   Source 多个 Package 段；FlagGems 已按此设计三包脚手架（Phase 1
   暂时合并为 python3-flag-gems，Phase 2 拆分）。

## 六、本次最大的工程教训

- **`COPY . /src` 是缓存毒药**：任何 packaging/ 编辑都让 36 分钟的
  wheel 构建从头来。已在 FlagTree 的 Dockerfile 中收窄。
- **`tail -N` 会吞错误**：早期 FlagTree 构建表面成功实则失败，因为
  `cmd 2>&1 | tail -200` 让 RUN 始终返回 0。已替换为 `set -o pipefail`。
- **网络下载是脆弱点**：anaconda.org SSL EOF 让本会话至少损失了 3 次
  完整构建。下一步预置 tarball 可消除这条故障路径。
- **Wheel 的 Python ABI 是硬约束**：cp310 wheel 无法在 Python 3.14
  主机上 install。一份 wheel 不能覆盖所有目标发行版，需建多 Python 矩阵。
